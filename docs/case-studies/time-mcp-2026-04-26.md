# `modelcontextprotocol/servers` MCP audit — `src/time` (2026-04-26)

> **Lite-scan audit** of `mcp-server-time@0.6.2` (2 tools, 0 prompts, 221 LOC Python in `src/time/`). **Zero security findings** across SEC-001..SEC-009 — the server performs only local datetime computation with no network, filesystem, or database access. One **Low** finding: no MCP-layer integration test (unit tests are comprehensive with ~30 test cases including DST transitions and fractional-offset edge cases). ## 2. Methodology

**Audit variant: lite scan** (§6.5 proposal from [retro #10](../../docs/retrospectives/first-external-audit-2026-04-26.md)). For servers with <10 tools, the full 5-axis rubric is overkill — three axes (return-shape, naming, decay) are trivially satisfied. The lite-scan runs only:

1. **Inventory + git-history scan** — tool list, security-keyword commit search, unmerged branch check. Pickaxe discipline mandatory per wave-2 correction.
2. **Security axis** (SEC-001..SEC-009 from SEVERITY.md) — the axis most likely to find actionable issues on any surface size.
3. **Test coverage axis** — MCP-layer integration test presence + unit test assessment.

Skipped axes and rationale:
- **Discoverability**: 2 tools, both well-described in README with example interactions. Would score 4-5/5 based on quick read — not worth a formal scoring table.
- **Return-shape consistency**: Both tools return `list[TextContent]` with `json.dumps(result.model_dump())`. Trivially consistent (N=2, same shape).
- **Naming**: `get_current_time`, `convert_time` — clear verbs, no prefix needed at N=2.
- **Decay**: Active server, both tools documented in README, tested, git activity within 90 days.

Severity bucketing uses [`SEVERITY.md`](../../skills/mcp-audit/SEVERITY.md). No overrides needed.

**Pickaxe discipline**: `git log --all -- src/time/` was scanned for security keywords (`secur`, `block`, `allow`, `auth`, `restrict`, `sanitize`, `validate`, `vuln`, `ssrf`, `inject`). Zero matches. Two unmerged branches found (`adamj/add-time-mcp-servers`, `adamj/mcp-server-time-url`) — both are community-server listing / registry metadata, not security-related. Verified with `git log --oneline <branch> -3`.

### F-001: No MCP-layer integration test (Low)

**Category**: TEST-001

**Evidence**: `test/time_server_test.py` contains ~30 tests across 5 logical groups:

| Group | Count | What's tested |
|---|---|---|
| `test_get_current_time` (parametrized) | 6 | UTC offsets, DST on/off, 3 timezones |
| `test_get_current_time_with_invalid_timezone` | 1 | McpError on invalid tz |
| `test_convert_time_errors` (parametrized) | 3 | Invalid source tz, invalid target tz, invalid time format |
| `test_convert_time` (parametrized) | 15 | DST transitions, fractional offsets (Nepal +5:45, Iran +3:30, Chatham +13:45), date-line crossings (Samoa, Kiritimati), historical offsets (Venezuela 2016), variable DST (Israel), Antarctica/Troll |
| `test_get_local_tz_*` | 6 | Override, invalid, mock tzlocal, Windows tz names, various IANA names |

All tests exercise `TimeServer()` methods directly using `freezegun` for deterministic time. The edge-case coverage is notably thorough (15 parametrized conversion cases covering real-world timezone oddities).

**What's missing**: No test instantiates an MCP `Client`, connects via `StdioClientTransport`, and calls the tools through the MCP protocol layer. This leaves untested:
- The `@server.call_tool()` dispatch (match/case on tool name, argument extraction)
- Input validation at the MCP boundary (missing timezone arg → ValueError)
- JSON serialization of `model_dump()` output through the MCP content layer
- The `@server.list_tools()` schema definitions

**Mitigating factors**: The MCP dispatch layer is trivial (~30 LOC, a match/case that delegates to the same methods the unit tests cover). The serialization is `json.dumps(result.model_dump())` — Pydantic's well-tested serializer. The risk of a gap-induced regression is low.

**Not filing upstream**: The maintainer has invested heavily in unit-level edge-case coverage. An MCP integration test would add marginal value here. This is noted as a finding for completeness, not as a recommended action.

## 6. Reusable patterns — lite-scan validation feedback

### §6.5 validation: did the lite scan produce the right calibration?

**Question**: Was the lite scan the correct choice for this surface?

**Answer**: Yes. The full 5-axis scan would have added ~3 extra sections (discoverability table, return-shape matrix, decay signal matrix) that would all trivially pass:
- Discoverability: both tools have clear descriptions with IANA timezone examples + local-tz hint. Would score 4-5/5.
- Return-shape: both tools return `list[TextContent]` with JSON-serialized Pydantic model. 100% consistent.
- Naming: `get_current_time`, `convert_time` — standard verb + noun, no collision.
- Decay: both tools in README, tested, git-active.

The lite scan took ~10 minutes of analysis vs ~25+ for a full scan. The security axis found zero findings (expected for a computation-only server). The test-coverage axis found the same pattern as the fetch audit (unit-tested but no MCP-layer round-trip). The difference: fetch's unit tests cover only helper functions and miss business logic at the MCP boundary; time's unit tests cover the actual business logic thoroughly, making the MCP-layer gap much narrower.

**Conclusion**: Lite scan was correctly calibrated. The 3 skipped axes would have been noise (trivially passing with no actionable output).

### Question: were the skipped axes actually noise?

| Skipped axis | Would it have found anything? | Evidence |
|---|---|---|
| Discoverability | No — tools have trigger phrases, IANA examples, local-tz default hint | Quick README + description read confirms 4-5/5 |
| Return-shape | No — trivially consistent at N=2, both use `json.dumps(model_dump())` | Source read confirms same pattern |
| Naming | No — 2 clear verbs, no prefix ambiguity | Self-evident at N=2 |
| Decay | No — active maintenance, both tools in README + tests + git | `git log` shows activity within 90 days |

All 4 skipped axes confirmed as noise for this surface.

### New axis proposal for small surfaces?

Considered: "purpose clarity" axis (does the server's README explain when to use it vs alternatives?). For time, the README has example questions and interactions — purpose is clear. This axis would add value only for servers with overlapping capabilities (e.g., two different fetch-like servers). Not recommending it as a standard addition to the lite scan.

### Pattern: lite-scan N/A density as attack-surface signal

Of 9 SEC checks, 6 were N/A (no applicable attack surface). This high N/A ratio is itself a finding: it confirms the server is a pure-computation tool with no external-system interaction. For future lite scans, a >50% N/A rate on SEC checks could be an early signal to complete the scan quickly — the attack surface is minimal by construction.
