# `modelcontextprotocol/servers` MCP audit — git server (2026-04-26)

> Audited the `mcp-server-git` MCP surface (12 tools registered via the
> low-level `@server.list_tools()` + `@server.call_tool()` pattern). No
> Critical / High / Medium findings — security posture is strong (active
> defenses against path traversal, symlink escape, and flag injection,
> all with tests). Two **Low** findings (one error-idiom inconsistency in
> `git_branch`, one MCP-layer dispatch test gap) and one **Info** finding
> (in-code Tool() descriptions lack when-to-use + return-shape guidance,
> surface-wide). F-001 + F-003 batched into upstream
> [`modelcontextprotocol/servers#4048`](https://github.com/modelcontextprotocol/servers/pull/4048);
> F-002 deferred to next-session backlog.

## 2. Methodology

The audit applies the [`mcp-audit` skill](../../skills/mcp-audit/SKILL.md):
five axes scored per tool (discoverability, return-shape, naming, error
handling, test coverage), plus decay-candidate detection across four
signals (CHANGELOG / README / consumer-import / test).

Severity bucketing uses the default rubric in
[`skills/mcp-audit/SEVERITY.md`](../../skills/mcp-audit/SEVERITY.md).

Disclosure handling follows [`docs/disclosure-sop.md`](../disclosure-sop.md).

**Methodology adaptation — low-level python-sdk shape**: this server uses
the older `@server.list_tools()` / `@server.call_tool()` pattern, not
FastMCP's `@mcp.tool()` decorator. Inventory adapted accordingly:
identify tools from the `Tool(name=…, description=…)` entries in the
list returned by `list_tools()`, and trace dispatch through the `match`
in `call_tool`. The five-axis rubric still applies; the canonical
return shape on this surface is `[TextContent(type="text", text=…)]`,
not a `{ok, ...}` envelope — this is the SDK-canonical shape for
low-level python-sdk and was treated as such (no `SHAPE-001` deviation).

**Honest-scoring discipline**: scores were not graded on a curve. The
discoverability average came out at 3.0/5 — between WRG's 2.81/5 and
filesystem's 4.50/5 — because every description is informative WHAT but
none have when-to-use + return-shape doc. No findings were padded; no
findings were de-emphasised.

### 3.1 Critical
None.

### 3.2 High
None. Notably, the server has active defenses for every category that
typically lands in this bucket on a git-tooling surface:

- **Path traversal**: `validate_repo_path` resolves both paths and checks
  `relative_to`, with explicit handling for `..` traversal and symlink
  escape. Tested in `tests/test_server.py:278-315`.
- **Flag injection**: 6 tools (`git_diff`, `git_checkout`, `git_show`,
  `git_create_branch`, `git_log`, `git_branch`) reject inputs starting
  with `-` before passing them to `repo.git.*`. Each guard has a
  positive test plus a "malicious ref via filesystem manipulation" test.
- **Mutation gating**: the `--repository` CLI flag scopes all mutation
  operations to a single allowed repo; without it, `validate_repo_path`
  is a no-op (documented behaviour, not a finding).

### 3.3 Medium
None.

### 3.4 Low

#### F-001 — `git_branch` error idiom inconsistency
- **Where**: `src/git/src/mcp_server_git/server.py:285`
- **What**: `git_branch` returns `f"Invalid branch type: {branch_type}"`
  on invalid input (a normal-looking string). Every other helper in the
  module raises (`git_diff`, `git_checkout`, `git_show`,
  `git_create_branch`, `git_log` use `raise BadName/ValueError`).
- **Why this bucket**: SEVERITY row [`ERR-002`](../../skills/mcp-audit/SEVERITY.md)
  (mixed `raise` vs in-band string across surface). Default Low — no
  security impact, but the string return reaches the MCP client
  masquerading as a successful tool result, which is exactly the
  situation the rest of the surface avoids.
- **Suggested fix**: change to `raise ValueError(...)` with a message
  naming the valid options (`'local'`, `'remote'`, `'all'`); add a
  test in the same style as the existing flag-injection guard tests.
- **Disclosure**: included in [`#4048`](https://github.com/modelcontextprotocol/servers/pull/4048).

#### F-002 — No MCP-layer dispatch test (surface-wide)
- **Where**: `src/git/tests/test_server.py` — every test imports the
  helper functions directly (`from mcp_server_git.server import git_status, …`)
  and calls them with a `git.Repo` instance; none round-trip through
  the `@server.call_tool()` dispatcher with `(name, arguments)`.
- **What**: 12/12 tools have store-layer tests (good), but 0/12 are
  exercised through the actual MCP wrapper — the `match
  GitTools.<NAME>` dispatch, the `arguments` dict unpacking, the
  `[TextContent(type="text", text=…)]` wrapping. A regression in the
  dispatch table (e.g., a typo in a `case` clause) would not be caught.
- **Why this bucket**: SEVERITY row [`TEST-001`](../../skills/mcp-audit/SEVERITY.md)
  (no MCP-layer test for non-mutation tool), applied surface-wide.
  Default Low — failure would be observable to the caller; not a
  blast-radius issue.
- **Suggested fix**: add a small async fixture that constructs the `Server`
  + invokes `call_tool(name, arguments)` directly, then assert on the
  returned `TextContent[]`. One representative test per dispatch family
  (read / mutate / list) would cover the dispatcher round-trip. Estimate
  ~1-2 hr including fixture boilerplate.
- **Disclosure**: deferred. This is real Low coverage debt, but it
  needs async test fixture work that doesn't fit the docs+chore batch.
  Recommend the maintainer file a follow-up issue or accept this
  case-study as the public record.

### 3.5 Info

#### F-003 — In-code Tool() descriptions lack when-to-use + return-shape (surface-wide)
- **Where**: `src/git/src/mcp_server_git/server.py:309-440` — all 12
  `Tool(name=…, description=…)` entries in `@server.list_tools()`.
- **What**: descriptions are concise WHAT-statements ("Shows the
  working tree status", "Switches branches"). Per the discoverability
  rubric, they have 1 of 3 axes (WHAT) and lack the other two
  (when-to-use, return-shape doc). Per-tool score: 12 × 3/5. **Average:
  3.0/5**.

  README has structured Input/Returns blocks per tool, but agents
  reading the MCP tool surface only see the in-code description.
- **Why this bucket**: SEVERITY rows [`DISC-002`](../../skills/mcp-audit/SEVERITY.md)
  (missing when-to-use) + [`DISC-004`](../../skills/mcp-audit/SEVERITY.md)
  (missing return-shape doc), applied as one surface-wide finding
  (single-PR fix). Default Info.
- **Suggested fix**: rewrite each description to a consistent shape:
  WHAT (preserved) + when-to-use clause + return-shape hint. Adding
  the existing flag-injection guarantee to relevant tools' descriptions
  (`"...refs starting with '-' are rejected"`) makes the safety
  property discoverable from the tool surface, not only from source.
- **Disclosure**: included in [`#4048`](https://github.com/modelcontextprotocol/servers/pull/4048).

## 6. Reusable patterns

Patterns observed that should feed back into the audit infrastructure:

- **Pattern**: low-level python-sdk inventory adaptation
  - **Observed in**: every finding (the inventory itself adapted).
  - **Generalises because**: not every Python MCP server uses FastMCP's
    `@mcp.tool()`. The older `@server.list_tools()` + `@server.call_tool()`
    pattern is still common in the official `modelcontextprotocol/servers`
    monorepo. The five axes still apply; the inventory grep doesn't.
  - **Proposed action**: extend [`SKILL.md`](../../skills/mcp-audit/SKILL.md)
    with a brief "low-level python-sdk adaptation" subsection alongside
    the TS-adaptation subsection (added in the same skill update wave).
    Inventory rule: *"For low-level python-sdk, list tools from the
    `Tool(name=…)` entries in the list returned by `@server.list_tools()`,
    not from `@mcp.tool()` decorators. Trace dispatch through the
    `match` (or `if/elif`) in `@server.call_tool()`."*
  - **Tracked as**: optional follow-up; not blocking.

- **Pattern**: per-ecosystem canonical return shape, recurring
  - **Observed in**: F-003 reasoning (no SHAPE finding even though the
    surface doesn't use `{ok}`).
  - **Generalises because**: this is the third surface (filesystem TS,
    instinct Python, now this) where the canonical shape was *not*
    `{ok, ...}` but was internally consistent. The skill discipline of
    "don't force-fit `{ok}` onto a surface that has its own canonical
    shape" was correctly applied each time.
  - **Proposed action**: SKILL.md already covers this implicitly via
    the TS-adaptation subsection. No change needed; this is a
    confirmation that the discipline is working.
  - **Tracked as**: confirmed pattern, no action.

- **Pattern**: pre-existing security maturity → no findings
  - **Observed in**: §3.2 "High" being empty despite a git surface
    with mutation tools.
  - **Generalises because**: when a project has a pattern of accepting
    security-hardening PRs (this repo's `#3545` and `#2970`), the audit's
    Critical/High buckets stay empty. The audit then becomes a
    consistency / discoverability tool, not a security tool.
  - **Proposed action**: case-study format already accommodates this
    via the "no findings" honest-scoring rule. No change needed.
  - **Tracked as**: confirmed pattern, no action.

## Appendix B — methodology drift

- **Inventory adaptation**: skill SKILL.md says "grep for `@mcp.tool()`".
  This server uses `@server.list_tools()` returning `list[Tool]`. Adapted
  by reading the returned list directly. Documented as a reusable pattern
  in §6.
- **Return-shape canonical**: the surface uses `[TextContent(type="text", text=…)]`
  uniformly (12/12). Treated as the canonical shape for low-level
  python-sdk; no `SHAPE-001` finding raised. Per-ecosystem canonical
  shape recognition is consistent with the TS-adaptation precedent.
- **No CHANGELOG file**: provenance was traced via `git log -- src/git/`
  (recent commits #3589 added `git_branch`, #3545 added flag-injection
  guards across 4 tools). DECAY signal "CHANGELOG mention" was
  interpreted as "git history shows recent activity for this tool";
  no decay candidates flagged.
