# `modelcontextprotocol/servers` MCP audit — `src/fetch` (2026-04-26)

> Audited `mcp-server-fetch@0.6.3` (1 tool, 1 prompt, 289 LOC Python in `src/fetch/`). One **Medium** finding: IP blocking code was added in commit `a1226d4` (Aug 2025) then silently removed in `ec20ee7` (Aug 27 2025) during an unrelated null-values fix — an accidental security regression on a widely-used reference server. One **Low** (no MCP-layer integration tests), two **Info** (missing return-shape documentation, no CHANGELOG). Internal evidence record: [WRG#308](https://github.com/yakuphanycl/WinstonRedGuard/pull/308).

## 2. Methodology

The audit applies the [`mcp-audit` skill](../../skills/mcp-audit/SKILL.md): five axes scored per tool (discoverability, return-shape, naming, error handling, test coverage), plus decay-candidate detection across four signals (CHANGELOG / README / consumer-import / test).

Severity bucketing uses the default rubric in [`skills/mcp-audit/SEVERITY.md`](../../skills/mcp-audit/SEVERITY.md). One override is called out (F-001).

Disclosure handling follows [`docs/disclosure-sop.md`](../disclosure-sop.md). The Medium finding (F-001) was escalated to the audit coordinator (Agent A) before public disclosure per SOP section 2.

**Honest-scoring discipline**: this is a 1-tool server. Three of the five audit axes (return-shape consistency, naming consistency, decay detection) are trivially satisfied when there is only one tool. The audit still produced value through the test-coverage axis and through the security regression discovery, which is outside the standard 5-axis framework but surfaced during git-history review in the inventory step.

**Framework adaptation**: this server uses the low-level `mcp.server.Server` API, not FastMCP's `@mcp.tool()` decorators. Tool registration is via `@server.list_tools()` + `@server.call_tool()` handler pattern. The skill's grep command (`@mcp.tool()`) does not match; the inventory was done by reading the source directly. The 5-axis rubric applies without further adaptation.

**Skill threshold check**: SKILL.md says "Don't trigger when: Server has <10 tools." With 1 tool, the full 5-axis rubric is overkill. However, the security regression (F-001) would not have been found without the inventory + git-history step. Recommendation: for servers with <10 tools, run a lightweight "security + test scan" variant rather than skipping entirely.

### F-001: IP blocking regression (Medium)

**Category**: SEC-004 (Arbitrary URL fetch with no allowlist — SSRF)

**Default severity**: High

**Assigned severity**: Medium — `OVERRIDE: de-escalated`

**Why this bucket**: The README explicitly warns users: "This server can access local/internal IP addresses and may represent a security risk." The SSRF surface is by-design for the tool's purpose (fetching arbitrary URLs for an LLM). However, IP blocking code *was* implemented and then accidentally removed, making this a regression rather than a deliberate design choice. De-escalated from High to Medium because: (a) the README warning exists, (b) exploitation requires the MCP client to be on a network with reachable internal services, (c) the server respects `robots.txt` by default.

**Evidence**:

Commit `a1226d4` (Aug 17 2025, author: `claude[bot]`) added comprehensive IP blocking:

```python
# Default blocked IP ranges (local/internal networks)
DEFAULT_BLOCKED_IP_RANGES = [
    "127.0.0.0/8",      # Loopback
    "10.0.0.0/8",       # Private Class A
    "172.16.0.0/12",    # Private Class B
    "192.168.0.0/16",   # Private Class C
    "169.254.0.0/16",   # Link-local
    "224.0.0.0/4",      # Multicast
    "::1/128",          # IPv6 loopback
    "fc00::/7",         # IPv6 unique local
    "fe80::/10",        # IPv6 link-local
]
```

Plus `is_ip_in_ranges()`, `check_url_safety()`, host allowlisting via `--allowed-hosts`, and CLI args for `--blocked-ip-ranges`.

Commit `ec20ee7` (Aug 27 2025, author: `claude[bot]`, message: "fix: handle null values for optional fetch parameters") removed **all** of this code — the `import ipaddress`, `import socket`, the blocked ranges constant, the safety functions, and the CLI args. The commit message mentions only null-value handling; the IP blocking removal appears accidental.

The current `server.py` (289 LOC) has no IP blocking, no host allowlisting, and no `--allowed-hosts` argument.

**Recommendation**: Re-introduce IP blocking. The original implementation in `a1226d4` is a reasonable starting point. File as a public GitHub issue with the regression evidence.

### F-002: No MCP-layer integration test (Low)

**Category**: TEST-001

**Evidence**: `tests/test_server.py` contains 14 tests across 4 classes:
- `TestGetRobotsTxtUrl` (6 tests) — pure function, no mock
- `TestExtractContentFromHtml` (3 tests) — pure function, no mock
- `TestCheckMayAutonomouslyFetchUrl` (5 tests) — async, mocked httpx
- `TestFetchUrl` (6 tests) — async, mocked httpx

All tests exercise **helper functions** directly. None instantiate an MCP `Client`, connect to the server via `StdioClientTransport`, and call `fetch` through the MCP protocol layer. This means:
- Input schema validation (Pydantic `Fetch` model via `inputSchema`) is untested at the MCP boundary
- The `@server.call_tool()` dispatch path is untested
- Content truncation logic (start_index / max_length pagination) is untested
- The `@server.get_prompt()` path is untested

**Recommendation**: Add a `tests/test_mcp_integration.py` with Client-Server round-trip tests, similar to `src/filesystem/__tests__/` pattern (adapted for Python: use `mcp.client` + subprocess transport).

### F-003: Missing return-shape documentation (Info)

**Category**: DISC-004

The `fetch` tool description says: "Fetches a URL from the internet and optionally extracts its contents as markdown." It does not describe what the response looks like (a `TextContent` with `"Contents of {url}:\n{content}"` prefix, with truncation markers and pagination instructions).

An agent calling `fetch` cannot predict:
- That the response has a `"Contents of {url}:"` prefix
- That truncated content ends with `<error>Content truncated. Call the fetch tool with a start_index of N...</error>`
- That `start_index` >= content length returns `<error>No more content available.</error>`

This is cosmetic (Info) because the tool works correctly and the `<error>` markers are self-documenting at runtime.

**Category**: DECAY-003

The `src/fetch/` package has no `CHANGELOG.md`. Version history is only in `pyproject.toml` (`version = "0.6.3"`) and git log. This is consistent with the other reference servers in this monorepo (none have per-package changelogs), so it's a monorepo-wide pattern rather than a fetch-specific gap.

## 6. Reusable patterns

### Pattern: git-history security regression detection

The standard 5-axis audit does not include git-history analysis. However, the inventory step's `git log --all -- src/<path>/` naturally surfaces commit messages that reference security features. When a security-sounding commit is found, diffing forward from that commit to HEAD reveals regressions.

**Trigger**: inventory `git log` shows a commit with keywords: `block`, `allow`, `security`, `auth`, `restrict`, `sanitize`, `validate`.

**Action**: `git diff <security-commit>..HEAD -- <file>` — check if the security code still exists.

This pattern found F-001 (IP blocking regression) which would not have been caught by any of the five standard axes.

### Pattern: low-level Server API adaptation

When the target uses `mcp.server.Server` (not FastMCP), the inventory grep changes:

```python
# FastMCP pattern (SKILL.md default)
grep -rn "@mcp.tool()" src/

# Low-level Server API pattern
grep -rn "@server.call_tool\|@server.list_tools\|@server.get_prompt\|@server.list_prompts" src/
```

The 5-axis rubric applies without further changes. The only difference is that tool metadata (name, description, inputSchema) is defined inside the `list_tools` handler rather than via decorators.
